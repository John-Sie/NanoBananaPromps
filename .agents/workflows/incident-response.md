description: 生產事故應變 — 從發現到止血到修復到事後檢討
---

你是專案的事故指揮官。執行此 workflow 的目標是：**在最短時間內恢復服務，並確保同類事故不再發生。**

**與其他 workflow 的關係**：
- `rollback.md` — 本 workflow 的子流程，負責回退 code/部署
- `bug-investigation.md` — 止血後的根因排查
- `feature-branch.md` — 修復 bug 的開發流程
- `hotfix.md` — 若修復極小且明確，可直推 main

> [!CAUTION]
> **事故期間以恢復服務為最高優先。** 不要花時間找根因，先止血。

---

## 事故等級定義

| 等級 | 定義 | 回應時間 | 範例 |
|------|------|----------|------|
| **P0 — Critical** | 核心服務完全不可用 | 立即 | 首頁 500、無法登入、資料遺失 |
| **P1 — High** | 核心功能嚴重受損 | 15 分鐘內 | 付款失敗、資料顯示錯誤 |
| **P2 — Medium** | 非核心功能異常 | 1 小時內 | 特定頁面 UI 錯位、非關鍵 API 錯誤 |
| **P3 — Low** | 輕微問題，不影響使用 | 下個工作日 | 文案錯字、非關鍵動畫異常 |

> P3 不需要走本 workflow，直接走 `feature-branch.md` 或 `hotfix.md`。

---

# 🔴 Phase 1 — 確認與分級（< 5 分鐘）

### 1a. 確認事故

```bash
echo "=== 最近的部署 ==="
git log --oneline -5
echo ""
echo "=== 最新 tag ==="
git tag --sort=-version:refname | head -3
```

確認：
- [ ] 問題是什麼？（錯誤訊息 / 使用者回報 / 監控告警）
- [ ] 影響範圍？（全部使用者 / 特定群體 / 特定功能）
- [ ] 什麼時候開始的？（與最近部署的時間吻合嗎？）
- [ ] 事故等級？（P0 / P1 / P2）

### 1b. 通知相關人員

> 若為團隊開發，立即通知（即使非工作時間）：
> - P0：所有相關工程師 + PM
> - P1：直接負責的工程師 + PM
> - P2：下次站會提出即可

---

# 🟡 Phase 2 — 止血（< 15 分鐘）

以**最快速度**恢復服務，不考慮完美解法。

### 選項 A：回退部署（最常見）

→ 執行 `rollback.md`

### 選項 B：Feature Flag 關閉

若問題出在特定功能且有 feature flag：
```bash
# 關閉 feature flag（依實際機制）
# 例：更新 env var、修改 config、dashboard toggle
```

### 選項 C：緊急 Hotfix

若問題原因明確且修復極小（如 typo、config 錯誤）：
→ 執行 `hotfix.md`

### 選項 D：流量切換 / 維護模式

若以上都不適用：
```bash
# 啟用維護模式（依實際機制）
# 例：修改 middleware 回傳 503 + 維護頁面
```

**止血完成確認**：
- [ ] 服務已恢復或進入維護模式
- [ ] 使用者不再受到影響

---

# 🟢 Phase 3 — 根因排查

> 服務已恢復後，才開始排查。不要在服務中斷時花時間查原因。

→ 執行 `bug-investigation.md`

重點關注：
- 最近的 code 變更（`git log`）
- 最近的 dependency 升級
- 最近的 DB migration
- 環境變數變更
- 第三方服務異常

---

# 🔵 Phase 4 — 修復與驗證

1. **建立修復分支** → `feature-branch.md`
2. **完整測試** → 確保修復有效且無副作用
3. **部署修復版本** → `merge-to-main.md` 或 `release.md`
4. **驗證**：
   - [ ] 問題已解決
   - [ ] 無新問題引入
   - [ ] 回退的功能已恢復（若適用）

---

# 📝 Phase 5 — 事後檢討（Post-Mortem）

**在事故解決後 24-48 小時內完成。**


---

## 額外規則

- **止血優先，完美其次** — 事故期間不追求最佳解法
- **不責備個人** — post-mortem 聚焦系統改善，不追究個人責任
- **所有行動都要記錄** — 事故期間的每個操作都要留下紀錄
- **不要在事故期間做大改動** — 止血用最小變更，大修復等服務穩定後再做
